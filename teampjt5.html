가장 많이 나오는 질문

"Q. 블렌더 노드 에디터에서 예쁘게 색을 섞고 빛이 나게 만들었는데, 유니티로 넘기니까 그냥 칙칙한 회색 큐브가 됐어요. 왜 그런가요?"

쉐이더(Shader)의 정확한 개념

쉐이더(Shader)란 3D 그래픽에서 화면의 각 픽셀이 어떤 색상으로, 어떻게 빛날지(또는 어두워질지)를 계산하는 '렌더링 연산 프로그램(코드)'입니다. 블렌더의 쉐이더 노드와 유니티의 쉐이더는 기반 시스템이 다르기 때문에 연산 규칙 자체가 호환되지 않습니다.

머티리얼 (Material)

3D 모델의 표면에 할당되는 데이터 컨테이너입니다. 색상 값, 텍스처 이미지 파일(.png), 투명도 수치 등의 데이터를 저장합니다.

쉐이더 (Shader)

머티리얼의 데이터를 바탕으로, 조명에 어떻게 반응할지(플라스틱처럼 반짝일지, 만화처럼 보일지, 스스로 발광할지)를 계산하는 수학적 규칙입니다.

올바른 VRChat 월드 제작 파이프라인

1. 블렌더 (형태와 텍스처링)
  • 메쉬(Mesh): 3D 모델의 뼈대를 만듭니다.
  • UV 전개3D 입체 도형의 표면을 2D 평면 이미지로 펼치는 작업입니다. 종이접기의 전개도를 만드는 것과 같습니다.: 텍스처를 입히기 위해 3D 표면을 2D로 펼칩니다.
  • 텍스처링 & 베이킹: 블렌더 내부의 그림자, 질감 연산 결과를 고정된 '2D 이미지 파일(.PNG)'로 구워냅니다. (이 과정을 위해 블렌더에서 텍스처 작업을 완료해야 합니다.)
2. 이원화된 데이터 전송

블렌더에서 만든 결과물은 두 가지 파일로 나뉘어 유니티로 전송됩니다.
1) .FBX 파일: 모델의 형태, UV 좌표 전달
2) .PNG 파일: 블렌더에서 구운 텍스처 이미지 전달

블렌더 쉐이더 노드의 빛 연산 규칙 자체는 유니티로 호환되지 않습니다.
3. 유니티 (최종 렌더링 조립)
  • 머티리얼 할당: 가져온 FBX 모델에 유니티 머티리얼을 생성해 입힙니다.
  • 쉐이더 설정: 렌더링 규칙을 VRChat 호환 쉐이더(예: lilToon)로 변경합니다.
  • 텍스처 연결: 블렌더에서 구워낸 .PNG 이미지를 쉐이더 슬롯에 연결하여 유니티 환경에서 질감과 발광을 완성합니다.
VRChat 호환 쉐이더 심층 분석 및 실무 가이드

유니티의 기본 쉐이더(Standard)는 VRChat 환경에서 최적화 문제나 렌더링 오류(특히 조명이 없을 때 까맣게 타버리는 현상)를 일으킬 수 있어 월드 제작에 거의 사용하지 않습니다. 목적에 맞춰 아래 두 가지 검증된 무료 쉐이더를 적절히 섞어 사용하는 것이 실무 표준입니다.

쉐이더 다운로드 및 VCC 적용 안내

과거에는 인터넷에서 파일을 받아 직접 넣었지만, 현재 VRChat 제작의 표준은 VCC(VRChat Creator Companion)를 통한 패키지 자동 관리(VPM) 방식입니다.

릴툰 (lilToon)
공식 문서
  1. 공식 문서(웹사이트)에 접속합니다.
  2. Add to VCC 버튼을 클릭하여 VCC 앱을 엽니다.
  3. VCC의 Manage Project 창 우측 목록에 추가된 lilToon 옆의 (+) 버튼을 눌러 프로젝트에 설치합니다.
포이요미 (Poiyomi)
공식 문서
  1. 포이요미 공식 깃허브 Releases 탭에 접속합니다.
  2. 가장 최근의 무료 버전 `.unitypackage` 파일을 다운로드합니다.
  3. 실행 중인 유니티 하단의 Project 창 빈 공간으로 다운받은 파일을 드래그 앤 드롭하여 임포트(Import) 합니다.

유니티 패키지(.unitypackage)란?

단순히 파일을 압축하는 일반 ZIP 파일과 다릅니다. 3D 모델(FBX), 텍스처 이미지(PNG), 그리고 이들이 유니티 내에서 연결된 머티리얼과 프리팹(Prefab)의 '연결 상태(Dependencies)'까지 통째로 보존하여 묶어주는 유니티 전용 압축 포맷입니다. 이사 갈 때 가구를 분해하지 않고 랩으로 둘러싸서 형태 그대로 옮기는 것과 같습니다.

실전: 패키지 내보내기(Export)와 가져오기(Import)

보내는 사람 (Export)
  1. 1
    최상위 폴더 우클릭

    자신의 작업물이 모두 담긴 고유한 이름의 최상위 폴더를 우클릭하고 메뉴에서 Export Package...를 클릭합니다.

  2. 2
    의존성 체크 확인

    팝업창이 뜨면 좌측 하단의 Include dependencies가 체크되어 있는지 확인합니다. (이 옵션이 켜져 있어야 연결된 텍스처와 재질이 끊어지지 않습니다.)

    Include dependencies
    Export
  3. 3
    저장 및 공유

    저장된 .unitypackage 파일을 팀 디스코드나 카톡으로 씬 조립 담당자에게 전송합니다.

조립 담당자 (Import)
  1. 1
    드래그 앤 드롭

    전달받은 .unitypackage 파일을 실행 중인 유니티 하단의 Project 창 빈 공간으로 드래그하여 넣거나 더블클릭합니다.

  2. 2
    파일 덮어쓰기 경고 검토

    Import 창이 뜰 때, 파일명 옆에 노란색 경고 아이콘(⚠️)이 떠 있다면 내 프로젝트의 기존 파일을 덮어씌워서 날려버린다는 뜻입니다! 충돌이 발생하면 Import를 취소하고 폴더명을 수정하라고 지시하세요.

  3. 3
    임포트 완료 및 배치

    Import 버튼을 누르면 상대방이 작업했던 폴더와 프리팹이 고스란히 들어옵니다. 프리팹을 씬에 올려 조립합니다.

폴더 및 경로 정리 시 절대 주의사항 (Overwrite 대참사 방지)

모든 팀원이 아무 생각 없이 기본 Assets/Models/ 폴더를 동일하게 사용하고 패키지를 합치면, 나중에 임포트한 사람의 파일이 이전 사람의 파일을 덮어씌워 날려버리는 대참사가 발생합니다. 남이 만든 메인 신전이 내 나무 모델링으로 교체되는 비극을 막으려면 아래 구조를 반드시 지켜야 합니다.

CRITICAL

대참사 방지: 지속적인 씬 조립과 압축 백업(버전 관리)

씬 조립을 마감 직전에 한 번에 몰아서 하려다가는 에러와 충돌로 프로젝트가 산산조각 날 수 있습니다. 특정 주차를 기다리지 말고, 매주 수업 시간이나 공강 시간에 지정된 '메인 PC'에 모여 수시로 패키지를 합치고 테스트하세요.

가장 확실한 수동 버전 관리법

Git이나 Plastic SCM 같은 전문 버전 관리 툴 사용이 익숙하지 않다면, 원시적이지만 가장 확실한 방법을 습관화하세요.

  1. 팀원의 패키지를 Import 하기 직전에 반드시 유니티를 완전히 종료합니다.
  2. 자신의 유니티 프로젝트 폴더 전체를 통째로 .zip 압축합니다.
  3. 압축 파일명에 날짜와 상태를 명시하여 구글 드라이브나 외장하드에 백업합니다.
    예: TeamProject_0513_Before_Plaza_Import.zip

앞서 안내한 대로 물리적인 씬 조립은 메인 PC에서 공강 시간 등을 활용해 꾸준히 병합을 진행해야 합니다. 이번 수업 시간의 메인 미션은 각자가 만든 구역을 합치기 전, 해당 구역에 어떤 상호작용(Logic)이 필요한지 리스트업하고 VRC 구현 가능성을 타진하는 '로직 보드' 작성 단계입니다.

VRChat 전용 스크립팅: 우동(Udon)의 가상 머신 구조

왜 C# 스크립트 대신 자체 가상 머신(Udon VM)을 쓸까?

일반적인 유니티 C# 코드는 컴퓨터의 파일 시스템(폴더)에 직접 접근하거나 시스템을 조작할 수 있습니다. 만약 누군가 악성 코드를 넣은 월드를 만들면 접속한 유저의 PC가 해킹당할 수 있죠. 이를 막기 위해 VRChat은 코드가 월드 밖으로 빠져나가지 못하게 가두는 '샌드박스(Sandbox)' 형태의 격리된 실행 환경인 Udon VM(Virtual Machine)을 구축했습니다.

1. 개발자 작성 Udon Graph / U#
2. Udon Compiler VRC 전용 안전 코드로 번역
3. VRChat Client VM(가상머신) 위에서 실행

이 Udon 컴파일러가 알아들을 수 있도록 우리가 로직을 작성해서 넘겨주는 방식은 크게 두 가지로 나뉩니다.

1. 노드 기반: Udon Graph (우동 그래프)

블렌더의 쉐이더 노드나 언리얼 엔진의 블루프린트처럼 시각적인 노드를 선으로 연결하여 로직을 구성하는 방식입니다. VRChat SDK가 기본으로 제공합니다.

  • 코딩을 전혀 모르는 기획자나 아티스트도 직관적인 접근 및 수정 가능
  • 기능이 조금만 복잡해져도 선이 미로처럼 꼬이는 일명 '스파게티 노드' 현상 발생
2. 스크립트 기반: UdonSharp (U#) 실무 표준

개발자들이 원래 쓰던 C# 코드를 텍스트로 작성하면, 이를 Udon 언어로 자동 번역해 주는 강력한 컴파일러 툴입니다.

  • 에셋 스토어나 BOOTH에서 다운받는 대부분의 고품질 기믹(거울 토글, 비디오 플레이어, 미니게임 등)은 모두 이 U#을 기반으로 제작됩니다.
  • 원활한 사용을 위해 VCC를 통해 프로젝트에 UdonSharp 패키지가 설치되어 있어야 합니다.
코딩 없는 인터랙션 구현: Udon Graph 기초 작성법 및 필수 노드
1. Udon Graph 셋업 순서
  1. 프로젝트(Project) 창 빈 공간 우클릭 Create > VRChat > Udon > Udon Graph Program Asset 생성
  2. 기능을 넣을 3D 오브젝트(예: 버튼 큐브)를 선택하고 인스펙터에서 Add Component > Udon Behaviour 추가
  3. 생성한 Program Asset 파일을 Udon Behaviour 컴포넌트의 Program Source 빈칸에 끌어다 놓기
  4. Open Udon Graph 버튼을 눌러 노드 에디터 창 열기
2. 실행 흐름(Flow)과 데이터(Data)의 완벽한 이해
실행 순서 (Flow 포트)

"언제 이 행동을 할 것인가?"를 결정하는 도미노입니다. 하얀색 화살표 선이 연결되지 않은 노드는 아무리 데이터가 완벽하게 채워져 있어도 절대 실행되지 않습니다. (좌에서 우로 흐름)

데이터 (Data 포트)

"무엇을 재료로 쓸 것인가?"를 의미합니다. 색상은 데이터의 종류를 뜻합니다. 노란색(오브젝트), 보라색(참/거짓), 초록색(좌표) 등 같은 색상끼리만 연결할 수 있습니다.

코딩을 몰라도 괜찮아요: AI 기반 UdonSharp (U#) 작성 3단계
  1. A
    ChatGPT / Claude에게 프롬프트 요청

    AI에게 코드를 요구할 때는 VRChat 환경에 맞는 '마법의 키워드' 두 가지를 반드시 포함해야 합니다.

    "유저가 큐브(버튼)를 클릭하면 조명이 켜지고 꺼지는 코드를 짜줘.

    단, 반드시 UdonSharpBehaviour를 상속받는 VRChat UdonSharp (U#) 코드로 작성해줘.
    조작할 조명 오브젝트는 유니티 인스펙터에서 직접 넣을 수 있게 public 변수로 열어줘."
    마법의 프롬프트 키워드 해설
    • 1. UdonSharp (U#) 코드로 작성해줘: 일반 유니티 코드는 VRChat에서 100% 에러를 냅니다. VRChat 전용 우동 가상머신(VM)이 이해할 수 있는 전용 문법으로 번역해 달라는 명령어입니다.
    • 2. public 변수로 열어줘: 이 명령어가 없으면 AI는 코드 내부에 특정 조명 이름을 하드코딩해버립니다. public으로 열면, 코딩을 전혀 몰라도 유니티 화면(인스펙터)에 조명을 드래그해서 끼워 넣을 수 있는 '빈칸(슬롯)'이 마법처럼 생성됩니다.
  2. B
    C# 스크립트 생성 및 연결

기획 구현 난이도 및 추천 접근법

하
기본 컴포넌트 활용

코딩이 전혀 필요 없는 VRChat 기본 제공 기능입니다. 클릭 몇 번으로 적용할 수 있어 가장 권장하는 방식입니다.

  • VRC Pickup: 집어 들고 던질 수 있는 물건 (컵, 공 등)
  • VRC Station: 앉을 수 있는 의자/소파 설정
  • VRC Mirror: 거울 (Global/Local 토글 권장)
중
단순 트리거 & 애니메이션

스위치나 콜라이더(충돌체)를 이용해 오브젝트를 끄고 켜거나, 애니메이션을 재생하는 직관적인 상호작용입니다.

  • 버튼 UI: BGM 켜기/끄기, 조명 색상 변경
  • 자동문: 가까이 다가가면 열리는 슬라이딩 도어
  • 순간이동: 포탈을 타면 특정 구역으로 이동(Teleport)
상
고급 로직 (코딩 필요)

UdonSharp(U#)을 이용해 직접 코드를 짜야 하는 영역입니다. 초보자는 직접 구현을 피하고, 에셋 스토어나 부스(BOOTH)에서 무료 프리팹을 다운받아 조립하는 것을 강력히 권장합니다.

  • 글로벌 동기화: 타이머, 점수판, 미니 게임 로직
  • 비디오 플레이어: 유튜브 URL을 재생하는 스크린

팀 미션: 로직 보드(Logic Board) 작성

우리 조 월드에 들어갈 모든 상호작용(인터랙션)을 피그잼에 리스트업하고, [트리거(조건)] + [액션(결과)] 형태로 명확하게 정리하여 담당자를 지정하세요. 이 문서를 바탕으로 강사의 순회 피드백이 진행됩니다.

조건 (Trigger) 결과 (Action)
[거울 버튼]을 누르면 거울 활성화 (Local 권장)
[비밀의 방]에 진입하면 조명이 붉게 변함 (Animation)
[마법 지팡이]를 집으면 손끝에서 파티클 재생 (Pickup)

로직 보드 작성이 어느 정도 마무리되었다면, 강사가 각 조를 순회하며 물리적인 공간 비율(스케일)과 기획의 실현 가능성을 심층 점검합니다. 점검 시 다음 항목들을 준비해 주세요.

1. 화이트박싱 스케일 점검

유니티에 임시로 배치한 큐브(벽, 바닥, 장애물)들의 크기가 적절한지 점검합니다.

아바타 기준 더미(Capsule) 배치 필수
통로가 너무 좁거나 넓지 않은지 확인

2. 로직 타당성 검토

피그잼에 작성한 로직 보드를 보고, 학생 수준에서 직접 코딩이 가능한지, 무료 에셋을 활용해야 하는지 방향을 잡아줍니다.

너무 복잡한 네트워크 동기화 기믹 커트
애니메이션 등으로 대체 가능한 대안 제시

3. 에셋 매칭 및 솔루션 처방

특정 구현을 막막해하는 조에게 검증된 해결책을 현장에서 즉석으로 매칭하여 제공합니다.
